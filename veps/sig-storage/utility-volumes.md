## Overview

This proposal introduces a new API mechanism within KubeVirt to define **hot-pluggable, maintainence-focused volumes** that are attached directly to the `virt-launcher` Pod, rather than being exposed to the Virtual Machine (VM) itself. These `utilityVolumes` will facilitate various out-of-band operations such as collecting backup output, performing diagnostic data collection like memory-dump, logs, and other future operational needs. This design generalizes the concept currently present in the `memorydump` volume source, providing a more versatile and extensible solution for similar dynamic, host-level needs.

## Motivation

KubeVirt currently offers the `memorydump` volume source, which, upon request, hot-attaches a PersistentVolumeClaim (PVC) to the `virt-launcher` Pod to store a memory dump of the VM. While powerful for its specific use case, there is a broader need for similar **temporary, hot-pluggable, host-level data collection** that doesn't involve mounting inside the VM's guest OS. For example:

  * **Backup Data Collection:** This proposal is a **required foundational component for the [Incremental Backup VEP](https://github.com/kubevirt/enhancements/blob/main/veps/sig-storage/incremental-backup.md)**. The Incremental Backup VEP necessitates a mechanism to **dynamically attach a PVC** to the `virt-launcher` Pod to collect the backup output directly from libvirt. This allows for efficient data transfer and management of backup artifacts.
  * **Logging/Telemetry Export:** Dynamically attaching a PVC to dump internal KubeVirt component logs, QEMU logs, or other diagnostic output generated by the `virt-launcher` process itself, to a dedicated volume for analysis.

Adding more volumeSource types alongside the existing `memorydump` field for these diverse uses would lead to an overloaded, less intuitive, and semantically ambiguous API. A new, general, and clearly separated mechanism for `virt-launcher`-scoped operations is the best solution (more on this on the [Alternatives](#alternatives) section)

## Goals

  * Introduce a new API field, `spec.utilityVolumes`, within `VirtualMachineInstance` and `spec.template.spec.utilityVolumes` in `VirtualMachine` to define hot-pluggable volumes mounted into the `virt-launcher` Pod without being mounted to the VM guest.
  * Enable these `utilityVolumes` to be **hot-attachable and detachable on demand** to a running `virt-launcher` Pod, mirroring the dynamic lifecycle of the `memorydump` operation.
  * Ensure the API design is extensible to support additional utility volume source types in the future without breaking compatibility.

## Non Goals

  * Automate the actions performed with these utility volumes (e.g., automatically triggering a backup script when a volume is attached). This design only provides the volume attachment mechanism; the user or an external controller would be responsible for triggering the operations that utilize these volumes.

## Definition of Users

  * **Kubernetes Administrators/Operators:** Who need to dynamically attach storage to a `virt-launcher` pod to collect system-level data or perform maintenance tasks on VMs from the Kubernetes host perspective.
  * **KubeVirt Developers/Maintainers:** For debugging KubeVirt components (e.g., `virt-launcher`, QEMU) and collecting diagnostics from the hypervisor layer via a dedicated volume.
  * **Backup/DR Solution Vendors:** Who need a reliable, KubeVirt-native way to access VM-related data or state directly from the `virt-launcher` process for backup or recovery operations, without requiring a guest agent. **This feature is critical for enabling efficient incremental backups in KubeVirt.**

## User Stories

* As a cluster administrator, I want to hot-attach a PVC to a virt-launcher pod to collect diagnostic information that persists beyond the VM's lifecycle.
* As a developer, I want to hot-attach a PVC to a virt-launcher pod to gain temporary write access to persistent storage from the Pod's environment.

## Design

The proposed design introduces a new top-level field, `spec.utilityVolumes`, in the `VirtualMachineInstanceSpec` (and `spec.template.spec.utilityVolumes` in `VirtualMachine`). This field will define **hot-pluggable volumes**, mounted directly into the `virt-launcher` Pod, not exposed to the VM guest. This clean separation ensures clarity and avoids confusion with guest-facing volumes defined under `spec.volumes`.

### Core Design Components

1.  **New API Field (`utilityVolumes`):**

      * Introduce `utilityVolumes []UtilityVolume` in `VirtualMachineInstanceSpec` (and `VirtualMachineSpec`).
      * `UtilityVolume` will be a new KubeVirt type that mirrors the structure of KubeVirt's native `Volume` type, containing a `name` field for identification and a an inlined `UtilityVolumeSource`, maintaining consistency with KubeVirt's existing volume API patterns.
      * `UtilityVolumeSource` will define different sources of utility volumes tailored for `virt-launcher` Pod operations, such as `backup`, `memorydump`, and others (with more types expected to be added in the future), each being its own struct typically using PersistentVolumeClaims with specific configuration fields.
      * `UtilityVolumes` will not have a corresponding entry in the `domain.devices.disks` or similar sections, explicitly indicating they are not for the VM domain.

2.  **Volume Hot-Attachment/Detachment Mechanism:**

      * **Hotplug Process:** `utilityVolumes` use KubeVirt's existing volume hotplug infrastructure. When added to the VM or VMI specification, they are automatically recognized as hotplug volumes and mounted as directories in the `virt-launcher` Pod.
      * **Mounting Behavior:** Unlike regular volumes that are mounted as filesystems containing a `disk.img` file served to the VM domain, `utilityVolumes` are mounted as filesystem directories directly accessible within the `virt-launcher` Pod environment. This allows data to be written directly to and read from these mounted directories without VM guest involvement.
      * **Management Options:** 
          * **VM-level:** Add to `spec.template.spec.utilityVolumes` for persistent volumes that survive VM restarts
          * **VMI-level:** Add to `spec.utilityVolumes` for temporary, on-demand operations. This action requires a controller with the necessary permissions.
      * **Lifecycle:** The `virt-controller` watches for specification changes and the `virt-handler` manages Pod volume mounting/unmounting accordingly.

3.  **Status Reporting:**

Utility volumes will be reported in the existing VMI `status.volumeStatus` field alongside regular volumes, using the current `VolumeStatus` struct without modifications.
      * **Target Field Handling:** For utility volumes, the `Target` field will be empty since these volumes are not presented to the VM guest.
      * **Standard Status Fields:** Utility volumes will use the existing status fields (`Phase`, `Reason`, `Message`, etc.) to report their lifecycle state.


### Migration Compatibility

**Design Principle:** Utility volumes are temporary operational tools attached via pod-level attachment pods and are incompatible with libvirt's migration mechanisms. **Consequently, these volumes will not migrate with the VM.**

**Migration Process:**
1. **Pre-Migration:** In the unlikely event that a utility operation is active at migration time, each utility type must be designed to handle this possibility, either aborting the operation or ensuring its completion before migration proceeds.
2. **During Migration:** Utility volumes remain attached to source VM. Migration controller filters utility volumes preventing the creation of attachment pods on destination node, namespace or cluster.
3. **Migration Completion:** Just before migration transitions to succeeded state, utility volumes are automatically removed from both VMI and VM specifications to prevent post-migration re-attachment.
4. **Post-Migration:** No utility volumes exist on destination node. Manual reattachment is possible if needed and storage and access mode allows (e.g., RWO PVCs may need time to detach from source node).

**Rationale:** This approach avoids complex migration logic while ensuring no functional loss, since utility volumes cannot operate during migration anyway (e.g., memory dumps impossible while VM memory is transferring, same for VM backups).

## API Examples

### Add a utility volume

#### Before:
```yaml
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: my-vm
spec:
  running: true
  template:
    spec:
      domain:
        # VM domain configuration
      volumes:
        # Existing volumes for the guest VM
        - name: system-disk
          containerDisk:
            image: kubevirt/fedora-cloud-container-disk-demo:latest
      # No utility volumes initially
```

#### After (adding a memoryDump utility volume):
```yaml
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: my-vm
spec:
  running: true
  template:
    spec:
      domain:
        # remains unchanged
      volumes:
        # remains unchanged
        - name: system-disk
          containerDisk:
            image: kubevirt/fedora-cloud-container-disk-demo:latest
      # NEW FIELD:
      utilityVolumes:
        - name: memorydump-target
          memoryDump:
            persistentVolumeClaim:
              claimName: memorydump-pvc
```

## Alternatives

Two approaches involving placing utility volumes within the existing `spec.volumes` list rather than creating a separate `utilityVolumes` field were considered:

**Option A:** Add individual volume source types (e.g., `backupVolumeSource`) alongside the existing `memoryDumpVolumeSource`
**Option B:** Create a single `utilityVolumes` volumeSource type that contains multiple utility volume types, deprecating the standalone `memoryDumpVolumeSource` field

* **Pros:**
    * **Minimal API Surface Changes:** Leverages the existing `spec.volumes` structure and volume processing infrastructure
    * **Consistent with Existing Pattern:** Follows the established `memoryDumpVolumeSource` precedent (Option A) or consolidates under existing VolumeSource patterns (Option B)
    * **Familiar to Users:** Uses the same volume management patterns users already know

* **Cons:**
    * **Semantic Overload:** The `spec.volumes` list is fundamentally designed for volumes presented to the guest VM (except of existing memorydump source). Adding utility-specific volume sources here would overload this concept and confuse users about what volumes are accessible inside vs. outside the VM.
    * **Volume Processing Logic Confusion:** Existing volume processing code in controllers assumes all volumes in `spec.volumes` are for guest VM attachment (disk devices, filesystems). Adding utility volumes here would require extensive conditionals throughout the codebase to differentiate between guest and utility volumes, making the code more complex and error-prone with Option A also doing that for each new type that can be added in the future..
    * **Live Migration Complexity:** The live migration logic processes all volumes in `spec.volumes` as part of the VM's state that may need to be migrated. Utility volumes should NOT migrate (they're Pod-specific), requiring special exclusion logic throughout the migration codebase and creating inconsistent migration behavior within a single volume list.
    * **Snapshot/Backup restore:** VM restore logic typically includes all volumes in `spec.volumes` as part of the VM's persistent state. Utility volumes would be required to be excluded (same as currently done for memory-dump), requiring additional complexity.
    * **API Structure Issues:**
        * **Option A:** API pollution with each new utility use case requiring a new top-level volume source type, leading to an ever-growing list of utility-specific volume sources mixed with guest volume sources.
        * **Option B:** Excessive layering of source types, leading to a complex and deeply nested API structure (e.g., a `volume` with a `volumeSource` of type `utilityVolume` containing a type-specific `utilityVolumeSource`).

**Conclusion:** The chosen design of a separate `utilityVolumes` field alongside the `volumes` list is superior:
- **Clear Separation:** Maintains a clean distinction between guest volumes (`spec.volumes`) and utility volumes (`spec.utilityVolumes`)
- **Unified Management:** Groups all utility-attached volumes under a single, purpose-built field that clearly indicates their shared characteristic of being mounted as directories in the virt-launcher Pod
- **API Clarity:** Avoids semantic confusion and excessive nesting, making the API intuitive for users to understand and manage


## Scalability
This feature depends on existing [volume hotplug machinery](https://kubevirt.io/user-guide/storage/hotplug_volumes/) which will ultimately be the bottleneck with regard to scale. Adding/removing hotplug volumes requires creating/deleting and maintaining long-running Pods which can limit system scalability.

However, `utilityVolumes` are designed for short-lived operations with a specific lifecycle pattern: attach → collect data/output → detach. The goal is to temporarily attach volumes for data collection (backups, memory dumps, diagnostics, etc) and then promptly detach them once the operation completes. This usage pattern means there should not be many utility volumes attached simultaneously per VM, significantly reducing the scalability impact compared to long-term volume attachments.

## Update/Rollback Compatibility

  * **API Evolution:** Adding a new top-level field `utilityVolumes` is an additive change to the `VirtualMachineInstance` and `VirtualMachine` CRDs. This ensures backward compatibility with existing KubeVirt API objects. Older KubeVirt components will simply ignore the new field.
  * **Upgrade Process:**
      * During an upgrade, older `virt-launcher` Pods (managed by pre-upgrade KubeVirt components) will not be aware of or include `utilityVolumes`.
      * New `virt-launcher` Pods, created or recreated by the updated `virt-controller`, will correctly include the `utilityVolumes` defined in the VMI/VM spec.
      * Hot-attachment/detachment operations involving these new `utilityVolumes` would only be possible after the cluster's KubeVirt components are fully upgraded to support the new API and logic.
  * **Rollback:** Rolling back KubeVirt to a version that does not support `utilityVolumes` would mean that the controller would no longer process volumes defined in this new API field. The underlying PVCs and their data remain intact, but the volumes will become inaccessible through the `virt-launcher`. After a rollback, there is no guarantee that such volumes will remain attached, and there will be no mechanism to reattach them using the `utilityVolumes` API.

## MemoryDump Volume Source Deprecation

The existing `memoryDump` volume source will be deprecated and replaced with the new `utilityVolumes.memoryDump` approach.

**Implementation Strategy:**
* **Alpha Phase**: The legacy `memoryDump` volume source is marked as deprecated. The memory dump controller pipeline is amended to internally use `utilityVolumes.memoryDump` instead of `volumes[].memoryDump`
* **User Interface**: The `virtctl memory-dump` command interface remains unchanged - users continue to use the same commands
* **Controller Migration**: New memory dump operations will create `utilityVolumes.memoryDump` entries instead of `volumes[].memoryDump` entries
* **Legacy Volume Handling**: Existing VMs with `memoryDump` entries in their `volumes` list will be transitioned by the controller to use utility volumes.

**Backward Compatibility:**
* **Existing VMs**: VMs with existing `volumes[].memoryDump` entries will continue to function until the volumes are naturally removed
* **Controller Support**: Memory dump controllers will detect and handle both legacy `volumes[].memoryDump` and new `utilityVolumes.memoryDump` during the deprecation period
* **API Validation**: Validation will prevent conflicts between legacy and new memory dump specifications

**Benefits of Migration:**
* **Consistency**: Memory dumps follow the same patterns as other utility operations
* **Implementation Clarity**: Memory dump operations are clearly separated from guest storage
* **Future Development**: All memory dump enhancements will use the utility volumes framework

  - Validation of the new `utilityVolumes` field for different utility volume types and their underlying volume sources
  - Proper handling of various utility volume configurations (backup, memorydump, etc.)
  - Error handling for invalid or malformed utility volume specifications
- **Controller Logic:**
  - `virt-controller` logic for parsing `utilityVolumes` and managing dynamic attachment/detachment

  - Lifecycle management through different phases (in progress, failed, completed)
- **Backward Compatibility:**
  - Transition logic from legacy to new `utilityVolumes.memoryDump` format
  - Validation prevents conflicts between legacy and new memory dump specifications

### End-to-End Tests

- **Volume Lifecycle:**
  - Hot-attachment and detachment of `utilityVolumes` to running `virt-launcher` Pods
  - Status reporting in VMI volume status reflects utility volume state correctly
- **Integration Testing:**
  - Compatibility with existing volume hotplug infrastructure
  - Memory dump functionality migration from legacy to utility volume approach - make sure existing tests work as expected.

### Migration Test Coverage

- **Utility Volume Migration Behavior:**
  - Verify no utility volume attachment pods are created on target node during migration
  - Verify no utility volume attachment pods are created post-migration due to specification removal
  - Verify utility volumes are removed from VM and VMI specs and volume status post-migration

## Implementation Phases

### Alpha

**Complete Implementation**: All functionality is implemented in the alpha phase:

  * **API Design**: The `utilityVolumes` field is introduced in the VMI/VM API. It supports utility-specific volume types (initially backup and memorydump, with extensibility for additional types) primarily using PersistentVolumeClaims with purpose-specific configuration fields and is inherently hot-pluggable.
  * **Feature Gate**: The functionality is protected behind a feature gate to ensure safe rollout and testing. `UtilityVolumes` feature gate disabled by default
  * **Controller Logic**: Complete `virt-controller` and `virt-handler` logic for hot-attaching and detaching `utilityVolumes` to/from a running `virt-launcher` Pod using a declarative approach by updating the VM/VMI spec.
  * **Status Reporting**: Utility volume attachment status is reflected in the VMI volume status.
  * **Migration Protection**: Ensure utility volumes are detached from the source VM and not automatically re-attached on the destination.
  * **Memory dump source deprecation**: Code transition to use utility volumes and marking the previous field as deprecated.
  * **Error Handling**: Comprehensive error handling and resilience for utility volume operations.
  * **Testing**

### Beta

- **Feature Gate**: `UtilityVolumes` feature gate enabled by default
- **Maturity Criteria**: Feature has been tested and proven stable in real-world scenarios for at least one release

### GA

- **Feature Gate Removal**: Remove the `UtilityVolumes` feature gate entirely
- **Production Readiness**: Feature meets all criteria for production deployment
- **Legacy Cleanup**: Complete deprecation of `memoryDump` volume source (planned for 3 releases after GA)
